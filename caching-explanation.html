Context Pruning vs. Prompt Caching

Design note for cpit — can we deterministically prune context for more intelligence without wrecking cache economics?

1. The hypothesis

A core thesis of cpit: we can intelligently and deterministically prune context to get more intelligence out of an LLM. Less noise in the window means sharper attention and better answers.

The headline risk: pruning context breaks prompt caching, and broken caching makes every subsequent inference dramatically more expensive. So the design question is: can we prune aggressively while keeping the cache warm?

The original idea: keep tool-call results at the end of the conversation so we can cache everything up to (but not including) them. Then if a later request changes, removes, or shrinks an old tool result, the cached prefix survives and we still get the pruning benefit. A follow-up variation: push all tool calls and results to the end.

2. How Anthropic prompt caching actually works

The one mental model that decides everything here: the cache is a prefix, not a set of blocks.

  • A request is serialized in a fixed order: tools → system → messages (messages in array order). The cache key is the literal byte prefix of that serialization up to a cache_control breakpoint.
  • On each request, Anthropic finds the longest previously-cached prefix that still byte-matches, and reads it at ~0.1× input. Everything after the first mismatch is reprocessed; writing the new segment costs 1.25× (5-min TTL) or 2× (1-hour TTL).
  • Up to 4 breakpoints. TTL is 5 min by default, refreshed on every cache hit (a hot prefix stays warm indefinitely); 1-hour TTL is opt-in and pricier.
  • Minimum cacheable segment ≈ 1024 tokens (2048 on Haiku).
The rule that governs everything Any byte change at position i invalidates the cache from i to the end of the prompt. There is no per-block "ignore this" mechanism. The cache is content-addressed by prefix, full stop.

A normal tool-use conversation is serialized like this:

tools ← stable, huge, rarely changes system ← stable messages[0] user "do X" messages[1] assistant text + tool_use A messages[2] user tool_result A messages[3] assistant text + tool_use B messages[4] user tool_result B messages[5] assistant final answer

3. The variations, and why they fail

Variation A — keep tool results at the end

Why it fails API pairing constraint: a tool_result must appear in the user message immediately after the assistant message containing the matching tool_use. You cannot defer results to the end — the Messages API validates the adjacency and rejects anything else.

Prefix causality: even if you could, a turn-3 result is "at the end" only on turn 3's request. By turn 20 it sits deep in the prefix of every later request — and turns 4–20 were generated conditioned on it. Editing or pruning it invalidates the cache from turn 3 onward regardless of where it physically sits. That is exactly the cache break we were trying to avoid.

Variation B — push all tool calls and results to the end

Why it fails Same pairing constraint: you can't collect a conversation's worth of tool_use blocks and defer their results. The only legal batching is parallel tool calls — many tool_use blocks in one assistant turn, all their tool_result blocks in the very next user turn.

And "the end" is the end only for the current inference step. The moment the agent takes another step, this step's results are frozen mid-prefix again. Pruning them later still breaks the cache from that point. Reordering never escapes prefix causality.
What the hypothesis got right The instinct — "isolate the volatile stuff so it doesn't poison the stable prefix" — is correct. It just can't be achieved by reordering history. It's achieved by layering the prompt around a cache breakpoint (next section).

4. The proposal: two layers split by a breakpoint

Stop thinking "reorder the history." Think "stable prefix + regenerable volatile tail, split by a breakpoint."

┌─────────────────────────────────────────────────────────┐ │ STABLE PREFIX │ │ tools → system → task → summarized reasoning │ │ (text only — no dangling tool_use) │ └─────────────────────────────────────────────────────────┘ ⟂ cache breakpoint (1-hour TTL) ┌─────────────────────────────────────────────────────────┐ │ VOLATILE TAIL │ │ this step's tool_use + tool_result, paired │ │ freely rewritten / pruned / rebuilt every turn │ └─────────────────────────────────────────────────────────┘
  • Everything after the last breakpoint can be rewritten, pruned, or rebuilt every turn — and you only re-bill the cheap tail. The expensive tools/system block is never invalidated.
  • To advance a step you do not append raw results to the stable prefix. You fold a compressed form into the stable narrative (a small, stable extension → one cheap write) and discard the raw output.
  • Tool results therefore never become permanent prefix. They live in a regenerable scratchpad behind the breakpoint. This is the legitimate realization of the original instinct.

Net effect: a prune costs one 1.25× write on the small tail, then every subsequent turn that shares the new prefix reads at 0.1×. The expensive layer rides a 1-hour cache and is effectively free after the first call.

Free win, available now Emit tool calls as parallel batches (one assistant turn, one result turn) instead of sequential round-trips. Fewer prefix-extension points, a cleaner volatile tail, fewer cache writes per task — independent of everything else here.

5. The actual hard problem (it isn't caching)

With the two-layer scheme, caching is the easy part — the strict-suffix rule solves it cleanly. The hard part is coherence.

The real risk An earlier assistant decision was generated conditioned on a tool result. Prune that result and the model loses the grounding for a decision still sitting in its history. The thesis lives or dies on what you summarize the result into, not on where you put it. Get the compression right and the cache economics fall out for free.

6. The user-facing toggle: cache vs. context

These two goals genuinely trade off, so expose them as a setting. The two-layer scheme is the substrate; the toggle tunes how aggressively the volatile tail is pruned and how often raw results are folded into the stable prefix.

ModeBehaviorCost profileBest for
Prioritize cache Append-only; never prune mid-history. Compact rarely, only at large thresholds and at breakpoint boundaries. Tolerate context bloat. Cheap per turn (mostly 0.1× reads), but larger input token counts; occasional big re-cache at compaction. Long iterative sessions on one task; latency-sensitive; cost-sensitive work where the model still does fine with a noisy window.
Prioritize context Aggressively summarize/prune the volatile tail every step; fold compressed results into the stable narrative frequently. More frequent 1.25× writes, but much smaller total token counts; potentially fewer retries / better outcomes. Complex reasoning where stale or bulky tool output measurably degrades quality or misleads the model.
Balanced (default) Stable prefix always protected behind a 1-hour breakpoint. Tail pruned on a moderate threshold; fold-into-prefix on step boundaries only. Predictable: one cheap write per compaction, cheap reads in between. The common case; safe starting point before a user tunes toward either pole.

Crucially, both poles sit on the same machinery. The tools/system layer is cached on a 1-hour TTL in every mode — the toggle never risks the expensive block. It only decides how chatty vs. how lean the tail is, and how often we pay the small write to compress it.

Bottom line You can't keep a stable prefix through a mid-history edit — that's fundamental, not an implementation gap. But you can make pruning a strict-suffix operation behind a breakpoint, so the expensive layer is never touched and pruning costs one small write. The intelligence win is then gated by summarization quality, not by cache economics.
